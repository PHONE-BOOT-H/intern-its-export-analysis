# 레포 규칙

이 레포를 쓰고 정리하는 기준. 참고한 표준:
cookiecutter-data-science(구조), Conventional Commits(커밋), art-of-readme(README),
AI 문체 탐지 자료(문체).

## 문체

- 이모지 안 쓴다.
- 짧은 말을 쓴다. "활용/수행/도출" 대신 "쓰다/하다/뽑다".
- 과장 형용사 금지. "강력한, 혁신적, 야무진" 같은 말 대신 숫자와 사실.
- 필러 금지. "결론적으로", "~라는 점에서", "주목할 점은" 빼고 바로 말한다.
- 굵게 표시는 정말 중요한 한 줄에만. 도배하지 않는다.
- 문단으로 될 건 문단으로. 불릿 도배 안 한다.
- 능동태. "분석이 수행되었다" 대신 "분석했다".
- 셋씩 나열하는 습관 버린다. 필요한 만큼만.
- 수치엔 출처, 기간, 한계를 붙인다. 근거 없이 단정하지 않는다.

## README

순서: 맨 위 한 줄 정의 → 무엇/왜 → 빠른 실행 → 구조 → 핵심 발견 → 한계.
필요한 만큼만 짧게 쓰고 세부는 docs/로 링크한다.
읽는 사람이 관련 없으면 빨리 판단하고 넘어가게 배려한다.

## 구조 (데이터 프로젝트 표준)

```
data/raw/        원본. 불변. 손대지 않고 git에도 안 올린다.
data/processed/  전처리 결과
src/             재사용 코드
notebooks/       탐색용 (번호_이니셜_설명 순, 제출 코드는 여기 안 둔다)
docs/            문서
reports/         결과물 (그림, 브리프)
```

raw 데이터는 절대 수정하지 않는다. requirements 버전은 고정한다.

## 커밋 (Conventional Commits)

형식: `type: 요약` (명령형, 소문자 시작, 끝에 마침표 없음). 왜 했는지는 본문에.

요약은 짧게, 한 것만. 부제·수식어·"— 설명" 꼬리 금지.
나쁨: `feat: 전략 브리프 뼈대 초안 — 확정 숫자 채우고 구멍 6개로 남은 작업 고정`
좋음: `feat: 브리프 뼈대 초안`

- feat  새 기능
- fix   버그 수정
- docs  문서
- refactor  코드 정리
- data  데이터 추가/갱신
- chore 잡일

예: `feat: ADB 낙찰 국적별 리그테이블 집계`

## 계속 정리

- 스크래치(explore_*, 임시 파일)는 역할이 끝나면 정리하거나 지운다.
- 한 파일은 한 역할. 죽은 코드는 남기지 않는다.
- 커밋 전에 레포를 한 번 훑어 지저분한 게 없는지 본다.
